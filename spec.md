# Minimal Kernel Subset

This document specifies the first kernel subset for the project.  The goal is to define a small fragment that can be implemented cleanly in Lean 4 and that still exercises the central machinery around inductive types.  The subset is intentionally narrower than full Lean 4, because a serious proof of concept depends more on a precise boundary than on premature coverage claims.

The subset follows the shape of Lean's core language described in the Lean reference manual: a dependently typed lambda calculus whose basic terms include universes, function types, variables, constants, applications, and `let` bindings.  It also follows the reference manual's account of inductive declarations: user input provides a type constructor and constructors, while the kernel derives a recursor and the corresponding computation rules.  The implementation in this repository uses that model as guidance, but the specification below is authoritative for the project.

## Scope

The first subset covers axioms, transparent and opaque definitions, theorem declarations, inductive blocks, low-level quotient primitives, core projections, literal expressions, structure metadata, declaration replay, and a first Lean declaration importer.  An inductive block contains one or more inductive declarations that share a universe context and a parameter telescope.  The subset supports level-polymorphic axioms, definitions, theorem declarations, and inductive blocks, parameters and indices on inductive types, dependent constructor-field telescopes, strictly positive constructor fields, generated recursor families, raw natural and string literals, and the reduction rules required to compute through those recursors.  It still omits higher-level quotient declarations, direct recursive definition import, unsafe declarations, and an importer for arbitrary Lean environment closures.

The subset now includes `Prop` as `Sort 0` for axioms, proposition-valued function types, proof irrelevance, and proposition-valued inductive declarations.  Ordinary data types begin at `Sort 1`, corresponding to Lean's `Type 0`.  Inductive result universes are predicative level expressions closed under the inductive declaration's universe parameters; each result level must be either `Prop` or definitely above `Prop` under every universe instantiation.  Proposition-valued recursors eliminate only to `Prop` unless the inductive satisfies the syntactic subsingleton test stated below.  The term language carries explicit universe arguments on constants, because generated recursors for data types and large-eliminating proposition recursors need a motive universe parameter in order to exist as ordinary primitive constants rather than as a typing special case.

## Terms and Declarations

Expressions have nine forms: bound variables, sorts, constants, literals, application, lambda abstraction, dependent function types, projection, and `let` bindings.  Bound variables use de Bruijn indices.  Constants carry explicit universe instantiations, and the environment stores the corresponding level-parameterized primitive types.  A constant application is well formed only when its universe instantiation has the arity prescribed by the declaration and each supplied level is closed under the current universe context.  User-facing axioms and definitions may bind explicit universe parameters.  A declaration body may mention only those level parameters, and raw kernel entry points continue to reject open universe levels when no universe context is present.

The literal forms are raw natural literals and string literals, matching Lean's `Expr.lit` constructors.  A natural literal has type `Nat`, and a string literal has type `String`.  Natural-literal typing requires the environment to contain `Nat`, `Nat.zero`, and, for positive literals, `Nat.succ`, because those are the declarations used by the computation rule.  String-literal typing requires the environment to contain `String`.  Raw natural literals reduce by the Lean kernel scheme: `0` unfolds to `Nat.zero`, and `n + 1` unfolds to `Nat.succ` applied to the raw literal `n`, so full normalization turns a natural literal into the corresponding constructor spine.  String literals are typed neutral literals in this subset.  The full Lean reduction of string literals through `String.ofList` and character lists remains outside the current specification.

The environment admits checked declaration records.  An axiom adds a constant with a type closed under its declared universe parameters.  A definition adds a constant with a type, value, and reducibility hint closed under its declared universe parameters, and the inferred value type must be definitionally equal to the declared type under that same universe context.  The supported hints are regular definitions with a stored height, abbreviations, and opaque reducibility hints.  These hints are declaration metadata in the current implementation; they do not prevent unfolding.  A transparent definition unfolds during conversion; an opaque declaration stores its checked value but does not unfold.  A theorem adds a constant whose type must be a proposition, and the supplied proof value must check against that type.  Theorem values remain stored in the declaration record for auditing, but conversion never unfolds a theorem constant to its proof.  An inductive block adds one type constructor for each block member, one constructor constant for each constructor, and recursor constants backed by one shared block recursor family.  A projection declaration adds an auxiliary constant whose checked value is a lambda over the structure parameters and the structure value, ending in a core projection expression.  Quotient primitives enter as primitive declaration records with checked types and explicit reduction metadata.  The kernel checks the derived constructor, recursor, projection, and primitive types before it admits them, so generated or primitive declarations do not bypass the ordinary well-formedness discipline.

An ordered declaration script is the kernel-facing form of a post-elaboration module fragment.  Each script entry is one of the declaration classes listed above, and script admission folds over the entries by calling the same checker used by the direct declaration API.  Later export parsers may build these scripts, but the trusted boundary remains the script-entry checker rather than the parser.

A kernel-style inductive declaration may provide type-former types and constructor types instead of source-shaped field telescopes.  This form records the universe parameters, the number of shared inductive parameters, one type-former type for each block member, and the fully elaborated constructor types.  Admission reconstructs the local inductive block by splitting each type-former type into shared parameters, indices, and a result sort, then splitting each constructor type into the repeated parameters, fields, and target.  The reconstructed block then follows the ordinary inductive admission path, including positivity, universe checks, projection-field computation, and generated constructor and recursor validation.

An exported declaration script may also contain generated-constructor and generated-recursor checks.  These entries do not add new constants.  They require the environment to contain the constructor or recursor regenerated by a previous inductive declaration, and they compare the exported universe parameters and type against the regenerated declaration up to alpha-equivalence.  Generated-recursor checks also compare the exported recursor metadata: block members, parameter count, index count, motive count, minor-premise count, each rule's constructor name and field count, and each exported rule right-hand side.  This mirrors the replay discipline used for exported Lean environments: inductive declarations are checked once, while generated declarations are verified against the kernel output.

Dependency-aware replay may process a finite collection of declaration entries whose order is not already topological.  Each entry exposes the names it defines and the constants mentioned in its types, values, constructor fields, projection metadata, and structure metadata.  Replay repeatedly admits entries whose external dependencies already exist in the environment, while allowing a declaration to refer to names it defines itself, as mutual inductive declarations do.  If no remaining entry can be admitted, replay rejects the collection with unresolved dependencies.

The Lean declaration importer translates Lean kernel data into declaration-script entries.  It maps Lean universe levels, expressions, literals, reducibility hints, safe axioms, safe definitions, theorem declarations, safe opaque definitions, quotient declarations, and inductive declarations into the local representation; generated constructor and recursor `ConstantInfo` records become replay checks against already generated local declarations.  It can also translate a finite `ConstantInfo` snapshot by grouping each inductive type former with its constructor records, emitting one kernel-style inductive declaration for the group, and emitting generated-declaration checks for the constructor and recursor records in the same snapshot.  The environment-closure extractor starts from root Lean names, follows constants mentioned by types, values, inductive metadata, constructor metadata, recursor metadata, and literal types, and then sends the resulting finite snapshot through the same importer.  The importer erases binder annotations, the `let` nondependence flag, and expression metadata, because those fields do not change kernel typing or conversion.  It rejects level metavariables, free variables, term metavariables, partial definitions, Lean mutual-definition declarations, incomplete inductive snapshots, inconsistent inductive groups, and unknown names in an environment closure, because the current local kernel has no corresponding trusted rule.

Trusted Lean replay is safe-only.  The importer rejects Lean records marked unsafe, including axioms, definitions, opaque definitions, inductive declarations, generated constructors, and generated recursors.  Theorem records do not carry an unsafe flag in Lean's declaration data, but theorem admission still checks that the proof has the declared proposition type and keeps the proof opaque to conversion.  A caller may construct local declaration scripts directly, but those scripts are then kernel-facing inputs rather than Lean import data; this specification assigns the safe-only rule to the Lean replay boundary.

## Contexts and Substitution

Runtime contexts are stored innermost binder first: de Bruijn index `0` refers to the head of the context, and index `i + 1` refers one binder farther out.  Source-level telescopes are written outermost first.  Binding a telescope as dependent function types therefore introduces the first listed binder as the outermost binder and the last listed binder as the innermost binder.

A binder type stored in a context is relative to the context that existed when the binder was introduced.  When the checker looks up de Bruijn index `i`, it lifts the stored binder type by `i + 1` before returning it, so the type lives in the current context.  This rule is part of the typing specification, not an implementation convenience.

Two telescope-binding operations follow from this convention.  Binding a dependent telescope assumes each binder type is already written in the context of the binders that precede it, so the operation inserts dependent-function binders in source order without rewriting those types.  Binding an independent telescope assumes every binder type was written in the same outer context and therefore lifts each later binder type by the number of earlier binders; the body is supplied in the final context and is not rewritten by that operation.  Type instantiation across a telescope applies the simultaneous-substitution rule below to each binder type, preserving telescope length and source order.

Simultaneous substitution is the primitive multi-substitution operation.  Given values `[v_0, ..., v_n]` ordered outermost to innermost, simultaneous substitution replaces de Bruijn index `0` with `v_n`, index `1` with `v_{n-1}`, and so on, while preserving variables below the active cutoff.  Under a binder, the cutoff increases and inserted values are lifted to avoid capture.  The kernel does not define target-schema or telescope instantiation by repeated one-variable substitution, because that would allow later substitutions to rewrite variables inside earlier inserted values.

## Inductive Fragment

An inductive block introduces one or more inductive types atomically.  All block members share the block universe parameters and parameter telescope; each member then declares its own index telescope, result universe, and constructors.  The singleton `addInductive` operation is a wrapper around block admission.  Each constructor field list is a telescope: the first field type is checked under the parameters, and each later field type is checked under the parameters and the earlier fields.  A constructor for an indexed inductive declares its result target explicitly as an application of the inductive type to the parameters and index terms, and those index terms may depend on constructor fields.

The result universe of an inductive type remains explicit in the first subset, but the kernel checks it instead of trusting it.  The result level may mention only the inductive declaration's universe parameters, and it must be either `Prop` or definitely positive.  When a data-valued inductive declaration has constructors, the universe of each parameter type and each constructor field type must be no larger than the declared result universe.  Proposition-valued inductives do not use that data-universe bound; their constructor-field universes affect only whether the generated recursor may eliminate into data.  Constructorless data inductives remain exempt from the data-field bound, matching the fact that they introduce no constructor arguments whose types must live inside the resulting sort.

A constructor target must apply the inductive type at the declaration's universe parameters.  A recursive occurrence in a constructor field must use those same universe parameters and the declared inductive parameters.  Generated helper targets remember the universe levels of the type they eliminate, so nested recursion through another polymorphic inductive uses that inductive's constructors at the target levels.  Iota reduction checks the constructor universe arguments against the instantiated recursor target levels before it applies a minor premise.

Recursive occurrences must be strictly positive.  A recursive occurrence may not appear to the left of an arrow.  It may appear in the codomain of an arrow, producing function-valued induction hypotheses, and it may appear as an argument to another inductive type constructor only when the corresponding parameter of that inductive has already been established as positive.  If a recursive occurrence appears inside another inductive type constructor, none of that nested inductive's parameter arguments may mention local variables from the current field context; this matches Lean's rejection of nested inductive parameters that contain local variables.  This makes the positivity check compositional across earlier inductive declarations and across the current block: `List Tree` is accepted because `List` is positive in its element parameter, while a type such as `BadParam Tree`, where the parameter occurs negatively inside `BadParam`, is rejected.  For a mutual block, positive-parameter facts are computed as a joint fixed point, and recursive occurrences headed by any block member count as open recursive targets.  The kernel analyzes constructor field types after beta, delta, and zeta normalization to a canonical term, so local `let` bindings and reducible abbreviations do not change whether a field is accepted.

The kernel derives a shared primitive recursor family for each inductive block.  The family has one target for each block member in declaration order, followed by the nested positive helper targets discovered while analyzing constructor fields.  A block-member target uses that member's recursor name, while a nested helper target uses the first block member's recursor name with Lean's indexed helper suffix, such as `T.rec_1`.  The target chosen by a recursor constant determines which type the recursor eliminates, with a motive that quantifies over the indices before the target term.  The same family includes helper targets for nested positive types such as `List Tree` and for recursive function fields such as `(n : Nat) → List Tree`.  Each helper target carries its own local telescope and target universe levels, and the generated motive and minor-premise types quantify over that telescope before the target term.  In a mutual block, every recursor uses the block target order for motives and minor premises; the recursor metadata selects which target the named recursor eliminates.  The family builder canonicalizes reachable target schemas before comparing them, trims unused trailing locals from helper targets, and therefore gives definitionally equal nested targets one helper recursor rather than parallel duplicates.  Schema instantiation is simultaneous across the root parameters and the local telescope, which matters when a helper target has explicit locals.  Data recursors carry one explicit universe parameter for the common motive sort followed by the inductive universe parameters.  Proposition-valued recursors carry that motive universe parameter exactly when the inductive has no constructors, or when it has one constructor and every constructor field satisfies one of two conditions: the field type is a proposition, or the field value is definitionally equal to a whole index argument in that constructor's declared target.  A data field whose type is an inductive parameter but whose value is not forced by the target does not qualify; allowing that field would permit extracting an unconstrained witness from an irrelevant proof.  Each constructor in each reachable target contributes one minor premise.  A minor premise binds the helper target locals when present, then all constructor fields in telescope order, then the induction-hypothesis arguments for strictly positive recursive fields in field order.  The induction-hypothesis shape follows the field: recursive fields at any target indices contribute the motive instantiated at those indices, function-valued fields contribute a function of hypotheses, and nested inductive fields contribute a hypothesis through the corresponding helper recursor.  A saturated recursor application reduces by iota when its target reduces to a constructor application and the constructor's universe arguments and declared result target are definitionally equal to the instantiated recursor target, including all index arguments.

## Quotient Primitives

The subset includes the low-level quotient constants `Quot`, `Quot.mk`, `Quot.lift`, `Quot.ind`, and `Quot.sound`.  The local syntax keeps the underlying type and relation explicit, so `Quot.{u} α r` forms a type in `Sort u` when `α : Sort u` and `r : α → α → Prop`.  `Quot.mk` injects an element into the quotient, `Quot.sound` proves equality of quotient elements whose representatives are related, and `Quot.ind` eliminates quotients into propositions.

`Quot.lift` eliminates a quotient into a non-dependent target type.  Its arguments are the underlying type, the relation, the target type, a function on representatives, a proof that the function respects the relation, and a quotient value.  The trusted quotient computation rule reduces `Quot.lift α r β f resp (Quot.mk α r a)` to `f a` after checking that the constructor uses the same quotient universe, underlying type, and relation as the eliminator.  The kernel does not add the higher-level `Quotient` setoid API in this subset.

## Projections

A core projection expression has the form `proj S i s`, where `S` is an inductive type name, `i` is a zero-based projection index, and `s` is the structure value.  In this subset, `S` must name an inductive declaration with exactly one constructor.  The inferred type of `s` must reduce to an application of `S` to its declared parameters and indices.  Projection indices range over constructor fields after removing fields whose values are forced by whole target indices.  A constructor field that appears inside a computed index, rather than as the whole index argument, remains projectable.  The projection result type is the selected constructor-field type after substituting the structure parameters and, for dependent fields, substituting earlier constructor fields either with their projections from `s` or with the target index that forces them.

Projection reduction is a kernel rule on the core projection expression.  If `s` reduces to the unique constructor of `S`, `proj S i s` reduces to the selected constructor field after the constructor parameters.  The rule checks that the projection name, field index, and constructor shape match before reducing.  Projection functions are ordinary checked declarations with projection metadata; their bodies reduce by delta to the core projection expression and then by projection reduction.

Projections from proposition-valued one-constructor inductives are allowed only when the selected field type is itself a proposition after the structure parameters and earlier projections have been substituted.  This matches the same integrity boundary as large elimination from `Prop`: a projection may not extract computational data from an irrelevant proof.

Structure eta is a conversion rule for non-recursive, data-valued, one-constructor inductives in the supported projection fragment.  A constructor application whose projectable fields are exactly the projections of a value `s`, and whose index-forced fields match the corresponding target indices, is definitionally equal to `s` when the constructor target matches the inferred type of `s`.  Recursive one-constructor inductives may use projections, but this subset does not apply eta to them.

Structure metadata records which one-constructor inductives came from structure declarations.  The metadata lists direct fields, their projection functions, direct parent structures, and subobject fields used to embed parents.  It does not add a typing rule: parent projections and field projections remain checked declarations, and their reduction still goes through the projection rules above.  The metadata supports inherited-field lookup and flattened field order for future export and notation work.

## Conversion and Typing

Typing follows the ordinary rules for a dependently typed lambda calculus.  `Sort u` has type `Sort (u + 1)`, so `Prop` has type `Type 0`.  A dependent function type `∀ x : A, B` lives in `Sort (imax u v)` when `A : Sort u` and `B : Sort v`.  The level former `imax u v` reduces to `0` when `v` is `0`, and it reduces to `max u v` when `v` is known to be a data universe.  An unresolved symbolic `imax` remains explicit rather than being approximated by `max`.  Primitive recursors type-check through those same ordinary rules once their explicit universe argument is supplied, because the kernel stores them as ordinary constants with ordinary dependent function types.  Conversion uses normalization-based definitional equality in the first implementation.

The conversion relation includes beta, delta, zeta, iota reduction, projection reduction, natural-literal reduction, quotient reduction, structure eta, and proof irrelevance.  Beta reduction substitutes an argument into a lambda body.  Delta reduction unfolds transparent definitions and projection functions, and it does not unfold opaque definitions or theorem declarations.  Zeta reduction substitutes the bound value of a `let`.  Iota reduction applies a saturated recursor case to a constructor target and recursively computes the induction hypotheses for recursive fields.  Projection reduction selects a field from a constructor-headed structure value.  Natural-literal reduction computes the raw natural literal scheme described above.  Quotient reduction computes the saturated `Quot.lift` and `Quot.mk` redex described above.  Structure eta compares a supported structure value with the constructor rebuilt from its projections.  Proof irrelevance treats two terms as definitionally equal when they infer the same normalized proposition type.  Unsaturated recursor and quotient primitive constants do not reduce, but they still type-check as ordinary constants.

## Deliberate Omissions

| Feature | Status in first subset | Reason |
| --- | --- | --- |
| Full Lean declaration model | Partial | The importer covers safe nonrecursive Lean declarations, finite `ConstantInfo` snapshots, root-name closure extraction from Lean environments, and generated constructor or recursor replay checks, but it does not yet reconstruct recursive definition compilation artifacts. |
| Higher-level quotient API | Omitted | `Quotient`, setoids, and derived quotient recursors can be specified on top of the low-level `Quot` primitives. |
| Full structure elaboration metadata | Partial | The kernel records structure fields, parent subobjects, inherited-field order, and checked projection declarations, but it does not yet import Lean's full structure-extension records. |
| Full string literal computation | Partial | String literals type-check as `String`, but this subset does not yet reduce them through `String.ofList` and character-list constructors. |

These omissions are not placeholders for undocumented behavior.  The kernel either implements a feature or rejects it.  Later versions can enlarge the specification once the current fragment has stable examples, tests, and a clearer path toward universes and propositions.
