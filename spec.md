# Minimal Kernel Subset

This document specifies the first kernel subset for the project.  The goal is to define a small fragment that can be implemented cleanly in Lean 4 and that still exercises the central machinery around inductive types.  The subset is intentionally narrower than full Lean 4, because a serious proof of concept depends more on a precise boundary than on premature coverage claims.

The subset follows the shape of Lean's core language described in the Lean reference manual: a dependently typed lambda calculus whose basic terms include universes, function types, variables, constants, applications, and `let` bindings.  It also follows the reference manual's account of inductive declarations: user input provides a type constructor and constructors, while the kernel derives a recursor and the corresponding computation rules.  The implementation in this repository uses that model as guidance, but the specification below is authoritative for the project.

## Scope

The first subset covers axioms, transparent and opaque definitions, theorem declarations, inductive blocks, low-level quotient primitives, and core projections.  An inductive block contains one or more inductive declarations that share a universe context and a parameter telescope.  The subset supports level-polymorphic axioms, definitions, theorem declarations, and inductive blocks, parameters and indices on inductive types, dependent constructor-field telescopes, strictly positive constructor fields, generated recursor families, and the reduction rules required to compute through those recursors.  It excludes the parts of Lean 4 that would force a larger theory before the inductive fragment is stable, such as reducibility hints beyond transparency, inherited structure metadata, and exported-declaration parsing.

The subset now includes `Prop` as `Sort 0` for axioms, proposition-valued function types, proof irrelevance, and proposition-valued inductive declarations.  Ordinary data types begin at `Sort 1`, corresponding to Lean's `Type 0`.  Inductive result universes are predicative level expressions closed under the inductive declaration's universe parameters; each result level must be either `Prop` or definitely above `Prop` under every universe instantiation.  Proposition-valued recursors eliminate only to `Prop` unless the inductive satisfies the syntactic subsingleton test stated below.  The term language carries explicit universe arguments on constants, because generated recursors for data types and large-eliminating proposition recursors need a motive universe parameter in order to exist as ordinary primitive constants rather than as a typing special case.

## Terms and Declarations

Expressions have eight forms: bound variables, sorts, constants, application, lambda abstraction, dependent function types, projection, and `let` bindings.  Bound variables use de Bruijn indices.  Constants carry explicit universe instantiations, and the environment stores the corresponding level-parameterized primitive types.  A constant application is well formed only when its universe instantiation has the arity prescribed by the declaration and each supplied level is closed under the current universe context.  User-facing axioms and definitions may bind explicit universe parameters.  A declaration body may mention only those level parameters, and raw kernel entry points continue to reject open universe levels when no universe context is present.

The environment admits checked declaration records.  An axiom adds a constant with a type closed under its declared universe parameters.  A definition adds a constant with a type and value closed under its declared universe parameters, and the inferred value type must be definitionally equal to the declared type under that same universe context.  A transparent definition unfolds during conversion; an opaque definition stores its checked value but does not unfold.  A theorem adds a constant whose type must be a proposition, and the supplied proof value must check against that type.  Theorem values remain stored in the declaration record for auditing, but conversion never unfolds a theorem constant to its proof.  An inductive block adds one type constructor for each block member, one constructor constant for each constructor, and one primitive recursor family for each block root.  A projection declaration adds an auxiliary constant whose checked value is a lambda over the structure parameters and the structure value, ending in a core projection expression.  Quotient primitives enter as primitive declaration records with checked types and explicit reduction metadata.  The kernel checks the derived constructor, recursor, projection, and primitive types before it admits them, so generated or primitive declarations do not bypass the ordinary well-formedness discipline.

An ordered declaration script is the kernel-facing form of a post-elaboration module fragment.  Each script entry is one of the declaration classes listed above, and script admission folds over the entries by calling the same checker used by the direct declaration API.  Later export parsers may build these scripts, but the trusted boundary remains the script-entry checker rather than the parser.

A kernel-style inductive declaration may provide type-former types and constructor types instead of source-shaped field telescopes.  This form records the universe parameters, the number of shared inductive parameters, one type-former type for each block member, and the fully elaborated constructor types.  Admission reconstructs the local inductive block by splitting each type-former type into shared parameters, indices, and a result sort, then splitting each constructor type into the repeated parameters, fields, and target.  The reconstructed block then follows the ordinary inductive admission path, including positivity, universe checks, projection-field computation, and generated constructor and recursor validation.

## Contexts and Substitution

Runtime contexts are stored innermost binder first: de Bruijn index `0` refers to the head of the context, and index `i + 1` refers one binder farther out.  Source-level telescopes are written outermost first.  Binding a telescope as dependent function types therefore introduces the first listed binder as the outermost binder and the last listed binder as the innermost binder.

A binder type stored in a context is relative to the context that existed when the binder was introduced.  When the checker looks up de Bruijn index `i`, it lifts the stored binder type by `i + 1` before returning it, so the type lives in the current context.  This rule is part of the typing specification, not an implementation convenience.

Two telescope-binding operations follow from this convention.  Binding a dependent telescope assumes each binder type is already written in the context of the binders that precede it, so the operation inserts dependent-function binders in source order without rewriting those types.  Binding an independent telescope assumes every binder type was written in the same outer context and therefore lifts each later binder type by the number of earlier binders; the body is supplied in the final context and is not rewritten by that operation.  Type instantiation across a telescope applies the simultaneous-substitution rule below to each binder type, preserving telescope length and source order.

Simultaneous substitution is the primitive multi-substitution operation.  Given values `[v_0, ..., v_n]` ordered outermost to innermost, simultaneous substitution replaces de Bruijn index `0` with `v_n`, index `1` with `v_{n-1}`, and so on, while preserving variables below the active cutoff.  Under a binder, the cutoff increases and inserted values are lifted to avoid capture.  The kernel does not define target-schema or telescope instantiation by repeated one-variable substitution, because that would allow later substitutions to rewrite variables inside earlier inserted values.

## Inductive Fragment

An inductive block introduces one or more inductive types atomically.  All block members share the block universe parameters and parameter telescope; each member then declares its own index telescope, result universe, and constructors.  The singleton `addInductive` operation is a wrapper around block admission.  Each constructor field list is a telescope: the first field type is checked under the parameters, and each later field type is checked under the parameters and the earlier fields.  A constructor for an indexed inductive declares its result target explicitly as an application of the inductive type to the parameters and index terms, and those index terms may depend on constructor fields.

The result universe of an inductive type remains explicit in the first subset, but the kernel checks it instead of trusting it.  The result level may mention only the inductive declaration's universe parameters, and it must be either `Prop` or definitely positive.  When a data-valued inductive declaration has constructors, the universe of each parameter type and each constructor field type must be no larger than the declared result universe.  Proposition-valued inductives do not use that data-universe bound; their constructor-field universes affect only whether the generated recursor may eliminate into data.  Constructorless data inductives remain exempt from the data-field bound, matching the fact that they introduce no constructor arguments whose types must live inside the resulting sort.

A constructor target must apply the inductive type at the declaration's universe parameters.  A recursive occurrence in a constructor field must use those same universe parameters and the declared inductive parameters.  Generated helper targets remember the universe levels of the type they eliminate, so nested recursion through another polymorphic inductive uses that inductive's constructors at the target levels.  Iota reduction checks the constructor universe arguments against the instantiated recursor target levels before it applies a minor premise.

Recursive occurrences must be strictly positive.  A recursive occurrence may not appear to the left of an arrow.  It may appear in the codomain of an arrow, producing function-valued induction hypotheses, and it may appear as an argument to another inductive type constructor only when the corresponding parameter of that inductive has already been established as positive.  This makes the positivity check compositional across earlier inductive declarations and across the current block: `List Tree` is accepted because `List` is positive in its element parameter, while a type such as `BadParam Tree`, where the parameter occurs negatively inside `BadParam`, is rejected.  For a mutual block, positive-parameter facts are computed as a joint fixed point, and recursive occurrences headed by any block member count as open recursive targets.  The kernel analyzes constructor field types after beta, delta, and zeta normalization to a canonical term, so local `let` bindings and reducible abbreviations do not change whether a field is accepted.

The kernel derives a primitive recursor family for each inductive root in a block.  The primary recursor eliminates that root, with a motive that quantifies over the indices before the target term.  The same family includes helper targets for other block members reached by mutual recursion, nested positive types such as `List Tree`, binder-dependent targets such as `(n : Nat) → WrapAt n Tree`, and targets that depend on earlier constructor fields.  Each helper target carries its own local telescope and target universe levels, and the generated motive and minor-premise types quantify over that telescope before the target term.  The family builder canonicalizes those reachable target schemas before comparing them, trims unused trailing locals from helper targets, and therefore gives definitionally equal nested targets one helper recursor rather than parallel duplicates.  Schema instantiation is simultaneous across the root parameters and the local telescope, which matters when a helper target depends on both.  Data recursors carry the inductive universe parameters and one explicit universe parameter for the common motive sort.  Proposition-valued recursors carry that motive universe parameter exactly when the inductive has no constructors, or when it has one constructor and every constructor field satisfies one of two conditions: the field type is a proposition, or the field value is definitionally equal to a whole index argument in that constructor's declared target.  A data field whose type is an inductive parameter but whose value is not forced by the target does not qualify; allowing that field would permit extracting an unconstrained witness from an irrelevant proof.  Each constructor in each reachable target contributes one minor premise, and each strictly positive recursive field contributes an induction-hypothesis argument immediately after the field that generated it.  The induction-hypothesis shape follows the field: recursive fields at any target indices contribute the motive instantiated at those indices, function-valued fields contribute a function of hypotheses, and nested inductive fields contribute a hypothesis through the corresponding helper recursor.  A saturated recursor application reduces by iota when its target reduces to a constructor application and the constructor's universe arguments and declared result target are definitionally equal to the instantiated recursor target, including all index arguments.

## Quotient Primitives

The subset includes the low-level quotient constants `Quot`, `Quot.mk`, `Quot.lift`, `Quot.ind`, and `Quot.sound`.  The local syntax keeps the underlying type and relation explicit, so `Quot.{u} α r` forms a type in `Sort u` when `α : Sort u` and `r : α → α → Prop`.  `Quot.mk` injects an element into the quotient, `Quot.sound` proves equality of quotient elements whose representatives are related, and `Quot.ind` eliminates quotients into propositions.

`Quot.lift` eliminates a quotient into a non-dependent target type.  Its arguments are the underlying type, the relation, the target type, a function on representatives, a proof that the function respects the relation, and a quotient value.  The trusted quotient computation rule reduces `Quot.lift α r β f resp (Quot.mk α r a)` to `f a` after checking that the constructor uses the same quotient universe, underlying type, and relation as the eliminator.  The kernel does not add the higher-level `Quotient` setoid API in this subset.

## Projections

A core projection expression has the form `proj S i s`, where `S` is an inductive type name, `i` is a zero-based projection index, and `s` is the structure value.  In this subset, `S` must name an inductive declaration with exactly one constructor.  The inferred type of `s` must reduce to an application of `S` to its declared parameters and indices.  Projection indices range over constructor fields after removing fields whose values are forced by whole target indices.  A constructor field that appears inside a computed index, rather than as the whole index argument, remains projectable.  The projection result type is the selected constructor-field type after substituting the structure parameters and, for dependent fields, substituting earlier constructor fields either with their projections from `s` or with the target index that forces them.

Projection reduction is a kernel rule on the core projection expression.  If `s` reduces to the unique constructor of `S`, `proj S i s` reduces to the selected constructor field after the constructor parameters.  The rule checks that the projection name, field index, and constructor shape match before reducing.  Projection functions are ordinary checked declarations with projection metadata; their bodies reduce by delta to the core projection expression and then by projection reduction.

Projections from proposition-valued one-constructor inductives are allowed only when the selected field type is itself a proposition after the structure parameters and earlier projections have been substituted.  This matches the same integrity boundary as large elimination from `Prop`: a projection may not extract computational data from an irrelevant proof.

Structure eta is a conversion rule for non-recursive, data-valued, one-constructor inductives in the supported projection fragment.  A constructor application whose projectable fields are exactly the projections of a value `s`, and whose index-forced fields match the corresponding target indices, is definitionally equal to `s` when the constructor target matches the inferred type of `s`.  Recursive one-constructor inductives may use projections, but this subset does not apply eta to them.

## Conversion and Typing

Typing follows the ordinary rules for a dependently typed lambda calculus.  `Sort u` has type `Sort (u + 1)`, so `Prop` has type `Type 0`.  A dependent function type `∀ x : A, B` lives in `Sort (imax u v)` when `A : Sort u` and `B : Sort v`.  The level former `imax u v` reduces to `0` when `v` is `0`, and it reduces to `max u v` when `v` is known to be a data universe.  An unresolved symbolic `imax` remains explicit rather than being approximated by `max`.  Primitive recursors type-check through those same ordinary rules once their explicit universe argument is supplied, because the kernel stores them as ordinary constants with ordinary dependent function types.  Conversion uses normalization-based definitional equality in the first implementation.

The conversion relation includes beta, delta, zeta, iota reduction, projection reduction, quotient reduction, structure eta, and proof irrelevance.  Beta reduction substitutes an argument into a lambda body.  Delta reduction unfolds transparent definitions and projection functions, and it does not unfold opaque definitions or theorem declarations.  Zeta reduction substitutes the bound value of a `let`.  Iota reduction applies a saturated recursor case to a constructor target and recursively computes the induction hypotheses for recursive fields.  Projection reduction selects a field from a constructor-headed structure value.  Quotient reduction computes the saturated `Quot.lift` and `Quot.mk` redex described above.  Structure eta compares a supported structure value with the constructor rebuilt from its projections.  Proof irrelevance treats two terms as definitionally equal when they infer the same normalized proposition type.  Unsaturated recursor and quotient primitive constants do not reduce, but they still type-check as ordinary constants.

## Deliberate Omissions

| Feature | Status in first subset | Reason |
| --- | --- | --- |
| Full Lean declaration model | Partial | Theorems and projection metadata are present for the first declaration fragment, while reducibility hints beyond transparent versus opaque definitions remain omitted. |
| Higher-level quotient API | Omitted | `Quotient`, setoids, and derived quotient recursors can be specified on top of the low-level `Quot` primitives. |
| Structure inheritance and parent projections | Omitted | The core projection rule now supports indexed one-constructor inductives, but inherited structure fields and parent projection generation require more declaration metadata. |

These omissions are not placeholders for undocumented behavior.  The kernel either implements a feature or rejects it.  Later versions can enlarge the specification once the current fragment has stable examples, tests, and a clearer path toward universes and propositions.
