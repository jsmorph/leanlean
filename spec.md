# Minimal Kernel Subset

This document specifies the first kernel subset for the project.  The goal is to define a small fragment that can be implemented cleanly in Lean 4 and that still exercises the central machinery around inductive types.  The subset is intentionally narrower than full Lean 4, because a serious proof of concept depends more on a precise boundary than on premature coverage claims.

The subset follows the shape of Lean's core language described in the Lean reference manual: a dependently typed lambda calculus whose basic terms include universes, function types, variables, constants, applications, and `let` bindings.  It also follows the reference manual's account of inductive declarations: user input provides a type constructor and constructors, while the kernel derives a recursor and the corresponding computation rules.  The implementation in this repository uses that model as guidance, but the specification below is authoritative for the project.

## Scope

The first subset covers axioms, transparent and opaque definitions, theorem declarations, inductive blocks, low-level quotient primitives, core projections, literal expressions, structure metadata, declaration replay, and a first Lean declaration importer.  An inductive block contains one or more inductive declarations that share a universe context and a parameter telescope.  The subset supports level-polymorphic axioms, definitions, theorem declarations, and inductive blocks, parameters and indices on inductive types, dependent constructor-field telescopes, strictly positive constructor fields, generated recursor families, raw natural and string literals, imported Lean structure-extension metadata, safe compiled recursive definitions, Lean kernel-overridden primitive reductions listed below, and the reduction rules required to compute through those recursors.  It still omits higher-level quotient declarations, source-recursion reconstruction, unsafe declarations, unspecified kernel primitive overrides, and broad replay of arbitrary Lean environment fragments.

The subset now includes `Prop` as `Sort 0` for axioms, proposition-valued function types, proof irrelevance, and proposition-valued inductive declarations.  Ordinary data types begin at `Sort 1`, corresponding to Lean's `Type 0`.  Inductive result universes are predicative level expressions closed under the inductive declaration's universe parameters.  A result level must be `Prop`, definitely above `Prop`, or one of the supported sort-polymorphic subsingleton shapes: no constructors, or one constructor with no fields.  A level is definitely above `Prop` when, after `imax` reduction and `max`/successor flattening, some summand has a positive successor offset; unresolved `imax` terms without such a summand do not satisfy the predicate.  This includes Lean's `PEmpty.{u}` and `PUnit.{u}` while preserving rejection for data-carrying `Sort u` declarations such as `BadPolyBox`.  Proposition-valued recursors eliminate only to `Prop` unless the inductive satisfies the syntactic subsingleton test stated below.  The term language carries explicit universe arguments on constants, because generated recursors for data types and large-eliminating proposition recursors need a motive universe parameter in order to exist as ordinary primitive constants rather than as a typing special case.

## Terms and Declarations

Expressions have nine forms: bound variables, sorts, constants, literals, application, lambda abstraction, dependent function types, projection, and `let` bindings.  Bound variables use de Bruijn indices.  Constants carry explicit universe instantiations, and the environment stores the corresponding level-parameterized primitive types.  A constant application is well formed only when its universe instantiation has the arity prescribed by the declaration and each supplied level is closed under the current universe context.  User-facing axioms and definitions may bind explicit universe parameters.  A declaration body may mention only those level parameters, and raw kernel entry points continue to reject open universe levels when no universe context is present.

This grammar describes post-elaboration kernel syntax:

```text
level   ::= 0 | param name | succ level | max level level | imax level level
literal ::= nat n | string s
expr    ::= bvar i
          | Sort level
          | const name [level*]
          | lit literal
          | app expr expr
          | lam name expr expr
          | forall name expr expr
          | proj structureName fieldIndex expr
          | let name expr expr expr
```

The literal forms are raw natural literals and string literals, matching Lean's `Expr.lit` constructors.  A natural literal has type `Nat`, and a string literal has type `String`.  Natural-literal typing requires the environment to contain `Nat`, `Nat.zero`, and, for positive literals, `Nat.succ`, because constructor-view computation refers to those declarations.  String-literal typing requires the environment to contain `String`.  Weak-head reduction and full normalization leave raw natural literals as literals, but conversion recognizes their numeric value and relates them to the corresponding `Nat.zero`/`Nat.succ` constructor spine.  A `Nat.rec` redex whose major premise is a raw literal views `0` as `Nat.zero` and `n + 1` as `Nat.succ n`, with the predecessor retained as a raw literal for the next step.  String literals are typed neutral literals in this subset.  The full Lean reduction of string literals through `String.ofList` and character lists remains outside the current specification.

The environment admits checked declaration records.  An axiom adds a constant with a type closed under its declared universe parameters.  A definition adds a constant with a type, value, and reducibility hint closed under its declared universe parameters, and the value must check against the declared type under that same universe context.  Lambda values are checked bidirectionally against declared dependent-function types, because inference alone can assign larger universe levels to nested motive functions than Lean records in checked declarations.  The supported hints are regular definitions with a stored height, abbreviations, and opaque reducibility hints.  These hints are declaration metadata in the current implementation; they do not prevent unfolding.  A transparent definition unfolds during conversion; an opaque declaration stores its checked value but does not unfold.  A theorem adds a constant whose type must be a proposition, and the supplied proof value must check against that type.  Theorem values remain stored in the declaration record for auditing, but conversion never unfolds a theorem constant to its proof.  An inductive block adds one type constructor for each block member, one constructor constant for each constructor, and recursor constants backed by one shared block recursor family.  A projection declaration adds an auxiliary constant whose checked value is a lambda over the structure parameters and the structure value, ending in a core projection expression.  Quotient primitives enter as primitive declaration records with checked types and explicit reduction metadata.  The kernel checks the derived constructor, recursor, projection, and primitive types before it admits them, so generated or primitive declarations do not bypass the ordinary well-formedness discipline.

An ordered declaration script is the kernel-facing form of a post-elaboration module fragment.  Each script entry is one of the declaration classes listed above, and script admission folds over the entries by calling the same checker used by the direct declaration API.  Later export parsers may build these scripts, but the trusted boundary remains the script-entry checker rather than the parser.

Script entries have the following abstract shapes:

| Entry | Required data | Admission rule |
| --- | --- | --- |
| Axiom | name, universe parameters, type | The type must be closed under the universe parameters and must infer a sort. |
| Transparent definition | name, universe parameters, reducibility hint, type, value | The type must infer a sort, the value must check against the declared type, and the constant unfolds during conversion. |
| Opaque definition | name, universe parameters, type, value | The type and value are checked as for a definition, but the constant does not unfold during conversion. |
| Theorem | name, universe parameters, proposition type, proof value | The declared type must live in `Prop`, the proof must check against it, and the theorem does not unfold during conversion. |
| Source-shaped inductive block | shared universe parameters, shared parameter telescope, member index telescopes, result levels, constructors, and targets | The block follows the inductive admission rule, including positivity, universe checking, generated declarations, and recursor-family construction. |
| Kernel-style inductive block | shared universe parameters, parameter count, type-former types, and constructor types | The checker reconstructs the source-shaped block and then applies the ordinary inductive admission rule. |
| Generated constructor check | name, universe parameters, exported type, parent inductive name | The exported type must match the constructor type regenerated by the local inductive checker. |
| Generated recursor check | name, universe parameters, exported type, optional recursor metadata | The exported type and metadata must match the recursor regenerated by the local inductive checker. |
| Projection declaration | name, structure name, projection index | The checker builds and checks a lambda ending in the corresponding core projection. |
| Structure metadata | structure name, field order, field projections, parents, and subobjects | The metadata must refer to checked projection declarations and already registered parent structures. |
| Quotient primitives | no additional data | The checker admits the specified low-level quotient primitive declarations after checking their types. |

A kernel-style inductive declaration may provide type-former types and constructor types instead of source-shaped field telescopes.  This form records the universe parameters, the number of shared inductive parameters, one type-former type for each block member, and the fully elaborated constructor types.  Admission reconstructs the local inductive block by splitting each type-former type into shared parameters, indices, and a result sort, then splitting each constructor type into the repeated parameters, fields, and target.  The reconstructed block then follows the ordinary inductive admission path, including positivity, universe checks, projection-field computation, and generated constructor and recursor validation.

An exported declaration script may also contain generated-constructor and generated-recursor checks.  These entries do not add new constants.  They require the environment to contain the constructor or recursor regenerated by a previous inductive declaration, and they compare the exported universe parameters and type against the regenerated declaration up to alpha-equivalence.  Generated-recursor checks also compare the exported recursor metadata: block members, parameter count, index count, motive count, minor-premise count, each rule's constructor name and field count, and each exported rule right-hand side.  This mirrors the replay discipline used for exported Lean environments: inductive declarations are checked once, while generated declarations are verified against the kernel output.

Dependency-aware replay may process a finite collection of declaration entries whose order is not already topological.  Each entry exposes the names it defines and the constants mentioned in its types, values, constructor fields, projection metadata, and structure metadata.  Replay repeatedly admits entries whose external dependencies already exist in the environment, while allowing a declaration to refer to names it defines itself, as mutual inductive declarations do.  Structure metadata has one extra readiness condition: a parent listed as a structure parent or subobject field must already be registered as structure metadata, not only present as an inductive constant.  If no remaining entry can be admitted, replay rejects the collection with unresolved dependencies.

The Lean declaration importer translates Lean kernel data into declaration-script entries.  It maps Lean universe levels, expressions, literals, reducibility hints, safe axioms, safe definitions, theorem declarations, safe opaque definitions, quotient declarations, and inductive declarations into the local representation; generated constructor and recursor `ConstantInfo` records become replay checks against already generated local declarations.  It can also translate a finite `ConstantInfo` snapshot by grouping each inductive type former with its constructor records, emitting one kernel-style inductive declaration for the group, and emitting generated-declaration checks for the constructor and recursor records in the same snapshot.  The environment-closure extractor starts from root Lean names, follows constants mentioned by types, values, inductive metadata, constructor metadata, recursor metadata, structure-extension metadata, and literal types, and then sends the resulting finite snapshot through the same importer.  For a Lean structure root, the extractor reads `Lean.getStructureInfo?`, adds direct field projection functions, parent projection functions, parent structures, and subobject structures to the closure, and emits a local structure-metadata declaration after the constants are available.  The structure importer validates Lean's projection-function metadata for each direct field against the structure constructor, parameter count, and field index before translation.  The importer erases binder annotations, the `let` nondependence flag, deprecated structure auto-parameter data, recursive-definition provenance metadata, and expression metadata, because those fields do not change kernel typing or conversion.  It rejects level metavariables, free variables, term metavariables, partial definitions, Lean mutual-definition declarations, incomplete inductive snapshots, inconsistent inductive groups, malformed structure-extension records, unsupported sort-polymorphic inductive results, and unknown names in an environment closure, because the current local kernel has no corresponding trusted rule.

Lean source-level recursive definitions are not kernel declarations.  Lean's declaration data records that fact: `DefinitionVal.all` is provenance information, while the compiled kernel declaration is an ordinary definition, and unsafe or partial recursion uses `mutualDefnDecl` or partial artifacts.  The environment importer therefore treats safe compiled recursive definitions as ordinary definitions: their types, values, and dependency closures must translate and recheck locally.  The importer still rejects `mutualDefnDecl` and partial definitions, and it does not reconstruct source recursion, equation metadata, structural-recursion choices, or well-founded-recursion proofs.

Trusted Lean replay is safe-only.  The importer rejects Lean records marked unsafe, including axioms, definitions, opaque definitions, inductive declarations, generated constructors, and generated recursors.  Theorem records do not carry an unsafe flag in Lean's declaration data, but theorem admission still checks that the proof has the declared proposition type and keeps the proof opaque to conversion.  A caller may construct local declaration scripts directly, but those scripts are then kernel-facing inputs rather than Lean import data; this specification assigns the safe-only rule to the Lean replay boundary.

## Contexts and Substitution

Runtime contexts are stored innermost binder first: de Bruijn index `0` refers to the head of the context, and index `i + 1` refers one binder farther out.  Source-level telescopes are written outermost first.  Binding a telescope as dependent function types therefore introduces the first listed binder as the outermost binder and the last listed binder as the innermost binder.

A binder type stored in a context is relative to the context that existed when the binder was introduced.  When the checker looks up de Bruijn index `i`, it lifts the stored binder type by `i + 1` before returning it, so the type lives in the current context.  This rule is part of the typing specification, not an implementation convenience.

Two telescope-binding operations follow from this convention.  Binding a dependent telescope assumes each binder type is already written in the context of the binders that precede it, so the operation inserts dependent-function binders in source order without rewriting those types.  Binding an independent telescope assumes every binder type was written in the same outer context and therefore lifts each later binder type by the number of earlier binders; the body is supplied in the final context and is not rewritten by that operation.  Type instantiation across a telescope applies the simultaneous-substitution rule below to each binder type, preserving telescope length and source order.

Simultaneous substitution is the primitive multi-substitution operation.  Given values `[v_0, ..., v_n]` ordered outermost to innermost, simultaneous substitution replaces de Bruijn index `0` with `v_n`, index `1` with `v_{n-1}`, and so on, while preserving variables below the active cutoff.  Under a binder, the cutoff increases and inserted values are lifted to avoid capture.  The kernel does not define target-schema or telescope instantiation by repeated one-variable substitution, because that would allow later substitutions to rewrite variables inside earlier inserted values.

## Inductive Fragment

An inductive block introduces one or more inductive types atomically.  All block members share the block universe parameters and parameter telescope; each member then declares its own index telescope, result universe, and constructors.  The singleton `addInductive` operation is a wrapper around block admission.  Each constructor field list is a telescope: the first field type is checked under the parameters, and each later field type is checked under the parameters and the earlier fields.  A constructor for an indexed inductive declares its result target explicitly as an application of the inductive type to the parameters and index terms, and those index terms may depend on constructor fields.

The result universe of an inductive type remains explicit in the first subset, but the kernel checks it instead of trusting it.  The result level may mention only the inductive declaration's universe parameters.  It must be `Prop`, definitely positive, an empty sort-polymorphic result, or a one-constructor no-field sort-polymorphic result.  The sort-polymorphic rule is intentionally narrow: it covers `PEmpty` and `PUnit`, and it rejects declarations whose `Sort u` result could become `Prop` while carrying unconstrained data.  For a data-valued inductive declaration, every constructor field after the shared inductive parameters must have a universe no larger than the declared result universe.  The shared parameters themselves are not subject to this bound; dependent family parameters such as `Î² : Î± â†’ Sort v` may live in a higher function-sort universe while the data introduced by constructors remains bounded by the result.  Proposition-valued inductives do not use the data-field bound; their constructor-field universes affect only whether the generated recursor may eliminate into data.  Constructorless data inductives remain exempt from the data-field bound, matching the fact that they introduce no constructor arguments whose types must live inside the resulting sort.

A constructor target must apply the inductive type at the declaration's universe parameters.  A recursive occurrence in a constructor field must use those same universe parameters and the declared inductive parameters.  Generated helper targets remember the universe levels of the type they eliminate, so nested recursion through another polymorphic inductive uses that inductive's constructors at the target levels.  Iota reduction checks the constructor universe arguments against the instantiated recursor target levels before it applies a minor premise.

Recursive occurrences must be strictly positive.  A recursive occurrence may not appear to the left of an arrow.  It may appear in the codomain of an arrow, producing function-valued induction hypotheses, and it may appear as an argument to another inductive type constructor only when the corresponding parameter of that inductive has already been established as positive.  If a recursive occurrence appears inside another inductive type constructor, none of that nested inductive's parameter arguments may mention local variables from the current field context; this matches Lean's rejection of nested inductive parameters that contain local variables.  This makes the positivity check compositional across earlier inductive declarations and across the current block: `List Tree` is accepted because `List` is positive in its element parameter, while a type such as `BadParam Tree`, where the parameter occurs negatively inside `BadParam`, is rejected.  For a mutual block, positive-parameter facts are computed as a joint fixed point, and recursive occurrences headed by any block member count as open recursive targets.  The kernel analyzes constructor field types after beta, delta, and zeta normalization to a canonical term, so local `let` bindings and reducible abbreviations do not change whether a field is accepted.

The kernel derives a shared primitive recursor family for each inductive block.  The family has one target for each block member in declaration order, followed by the nested positive helper targets discovered while analyzing constructor fields.  A block-member target uses that member's recursor name, while a nested helper target uses the first block member's recursor name with Lean's indexed helper suffix, such as `T.rec_1`.  The target chosen by a recursor constant determines which type the recursor eliminates, with a motive that quantifies over the indices before the target term.  The same family includes helper targets for nested positive types such as `List Tree` and for function-valued fields whose strictly positive occurrence passes through such a nested target, such as `(n : Nat) â†’ List Tree`.  A direct recursive occurrence headed by a block member, including one under function binders such as `Acc r y`, does not create a helper target; it records the occurrence target and instantiates the existing block-member motive at that occurrence's indices.  Each helper target carries its own local telescope and target universe levels, and the generated motive and minor-premise types quantify over that telescope before the target term.  In a mutual block, every recursor uses the block target order for motives and minor premises; the recursor metadata selects which target the named recursor eliminates.  The family builder canonicalizes reachable target schemas before comparing them, trims unused trailing locals from helper targets, and therefore gives definitionally equal nested targets one helper recursor rather than parallel duplicates.  It normalizes field types for positivity and helper-target discovery, but generated recursor declarations use the declared constructor field binders.  Schema instantiation is simultaneous across the root parameters and the local telescope, which matters when a helper target has explicit locals.  Data recursors and supported sort-polymorphic subsingleton recursors carry one explicit universe parameter for the common motive sort followed by the inductive universe parameters.  Proposition-valued recursors carry that motive universe parameter exactly when the inductive has no constructors, or when it has one constructor and every constructor field satisfies one of two conditions: the field type is a proposition, or the field value is definitionally equal to a whole index argument in that constructor's declared target.  A data field whose type is an inductive parameter but whose value is not forced by the target does not qualify; allowing that field would permit extracting an unconstrained witness from an irrelevant proof.  Each constructor in each reachable target contributes one minor premise.  A minor premise binds the helper target locals when present, then all constructor fields in telescope order, then the induction-hypothesis arguments for strictly positive recursive fields in field order.  The induction-hypothesis shape follows the field: recursive fields at any target indices contribute the motive instantiated at those indices, function-valued fields contribute a function of hypotheses, and nested inductive fields contribute a hypothesis through the corresponding helper recursor.  A recursor application reduces by iota when it contains the full recursor redex prefix, its target reduces to a constructor application, and the constructor's universe arguments and declared result target are definitionally equal to the instantiated recursor target, including all index arguments.  If the recursor result is itself a function and the application has trailing arguments after the major premise, iota reduction re-applies those trailing arguments to the reduced minor-premise result.

## Quotient Primitives

The subset includes the low-level quotient constants `Quot`, `Quot.mk`, `Quot.lift`, `Quot.ind`, and `Quot.sound`.  The local syntax keeps the underlying type and relation explicit, so `Quot.{u} Î± r` forms a type in `Sort u` when `Î± : Sort u` and `r : Î± â†’ Î± â†’ Prop`.  `Quot.mk` injects an element into the quotient, `Quot.sound` proves equality of quotient elements whose representatives are related, and `Quot.ind` eliminates quotients into propositions.

`Quot.lift` eliminates a quotient into a non-dependent target type.  Its arguments are the underlying type, the relation, the target type, a function on representatives, a proof that the function respects the relation, and a quotient value.  The trusted quotient computation rule reduces `Quot.lift Î± r Î² f resp (Quot.mk Î± r a)` to `f a` after checking that the constructor uses the same quotient universe, underlying type, and relation as the eliminator.  The kernel does not add the higher-level `Quotient` setoid API in this subset.

## Projections

A core projection expression has the form `proj S i s`, where `S` is an inductive type name, `i` is a zero-based projection index, and `s` is the structure value.  In this subset, `S` must name an inductive declaration with exactly one constructor.  The inferred type of `s` must reduce to an application of `S` to its declared parameters and indices.  Projection indices range over constructor fields after removing fields whose values are forced by whole target indices.  A constructor field that appears inside a computed index, rather than as the whole index argument, remains projectable.  The projection result type is the selected constructor-field type after substituting the structure parameters and, for dependent fields, substituting earlier constructor fields either with their projections from `s` or with the target index that forces them.

Projection reduction is a kernel rule on the core projection expression.  If `s` reduces to the unique constructor of `S`, `proj S i s` reduces to the selected constructor field after the constructor parameters.  The rule checks that the projection name, field index, and constructor shape match before reducing.  Projection functions are ordinary checked declarations with projection metadata; their bodies reduce by delta to the core projection expression and then by projection reduction.

Projections from proposition-valued one-constructor inductives are allowed only when the selected field type is itself a proposition after the structure parameters and earlier projections have been substituted.  This matches the same integrity boundary as large elimination from `Prop`: a projection may not extract computational data from an irrelevant proof.

Structure eta is a conversion rule for non-recursive, data-valued, one-constructor inductives in the supported projection fragment.  A constructor application whose projectable fields are exactly the projections of a value `s`, and whose index-forced fields match the corresponding target indices, is definitionally equal to `s` when the constructor target matches the inferred type of `s`.  Recursive one-constructor inductives may use projections, but this subset does not apply eta to them.

Structure metadata records which one-constructor inductives came from structure declarations.  The metadata lists direct fields, their projection functions, direct parent structures, and subobject fields used to embed parents.  Registration checks that each direct field projection is either a local projection declaration for the same structure and field position or a checked declaration whose stored value is a lambda ending in the corresponding core projection expression.  A parent projection must have a stored checked value, and its type must be a function from the child structure value to the declared parent structure after the child parameters are bound.  A parent marked as a subobject must have a matching subobject field with the same parent structure and projection function, and a referenced parent structure must already have registered structure metadata.  This metadata does not add a typing rule: parent projections and field projections remain checked declarations, and their reduction still goes through the projection rules above.  The metadata supports inherited-field lookup and flattened field order for future export and notation work.

## Conversion and Typing

Typing follows the ordinary rules for a dependently typed lambda calculus.  `Sort u` has type `Sort (u + 1)`, so `Prop` has type `Type 0`.  A dependent function type `âˆ€ x : A, B` lives in `Sort (imax u v)` when `A : Sort u` and `B : Sort v`.  The level former `imax u v` reduces to `0` when `v` is `0`, and it reduces to `max u v` when `v` is known to be a data universe.  The level equality rule also identifies `imax u u` with `u`, `imax 0 u` with `u`, and `imax 1 u` with `u`.  The last equation is valid because the right side is either `0`, where `imax` returns `0`, or a successor level, where `max 1 u` is `u`.  An unresolved symbolic `imax` remains explicit rather than being approximated by `max`.  Primitive recursors type-check through those same ordinary rules once their explicit universe argument is supplied, because the kernel stores them as ordinary constants with ordinary dependent function types.  Conversion computes weak-head normal forms and then compares syntax recursively, invoking conversion again on subterms.

Some Lean constants are ordinary declarations in export data but have kernel-overridden computation behavior.  A local checker that replays exported Lean terms must match those computation rules when Lean's kernel conversion used them, even if the source file also contains a logical model for the same constant.  This specification admits only named primitive reductions in the following table; adding another entry requires a source citation, a preservation argument, and tests that compare local conversion with Lean.  A listed rule is attached to the reserved kernel constant with the stated declaration shape, not to an arbitrary user declaration with the same printed name.

The conversion relation includes beta, delta, zeta, iota reduction, projection reduction, natural-literal numeric comparison, listed primitive reduction, quotient reduction, function eta, structure eta, and proof irrelevance.  Beta reduction substitutes an argument into a lambda body.  Delta reduction unfolds transparent definitions and projection functions, and it does not unfold opaque definitions or theorem declarations.  Zeta reduction substitutes the bound value of a `let`.  Iota reduction applies a recursor case to a constructor target and recursively computes the induction hypotheses for recursive fields.  Projection reduction selects a field from a constructor-headed structure value.  Natural-literal comparison identifies raw literals with constructor spines carrying the same numeric value without expanding large literals during normalization.  Listed primitive reduction applies only to the constants and arities stated in the primitive table.  Quotient reduction computes the saturated `Quot.lift` and `Quot.mk` redex described above.  Function eta identifies `Î» x, f x` with `f` when `f` has a dependent-function type and the lambda body is definitionally equal to applying `f` to the bound variable.  Structure eta compares a supported structure value with the constructor rebuilt from its projections.  Proof irrelevance treats two terms as definitionally equal when their inferred proposition types are equal by conversion.  Unsaturated recursor and quotient primitive constants do not reduce, but they still type-check as ordinary constants.

## Checker Interfaces

The executable checker API reports four outcomes.  `accepted` means that the artifact translated into the local declaration-script form and replayed through the local kernel.  `rejected` means that the artifact reached the local declaration-script boundary and violated a kernel rule, while `unsupported` means that the artifact used a Lean feature outside the specified importer or syntax.  `internal` means that the executable failed before it could assign one of the other meanings.

`leanlean-check-module` is a development checker for compiled Lean modules.  It loads a module with Lean's environment loader, takes explicit declaration roots, extracts the finite root-name closure, translates that closure through the Lean importer, and replays the resulting script locally.  This tool uses Lean's module loader and environment extensions, so it is a way to check local Lean programs against the current fragment rather than the independent `lean4export` checker interface.

Because `leanlean-check-module` consumes compiled environment data, it classifies a root by its loaded `ConstantInfo`, not by the source command that produced it.  A source-level `partial def` that Lean stores as a safe opaque constant therefore checks under the ordinary opaque-declaration rule.  Explicit partial declaration records in the export format remain unsupported.

`leanlean-self-check` is a development checker for this repository's compiled kernel modules.  Its default mode checks a named kernel-only slice, while `--module-closure` selects source-facing declarations from `LeanLean.Syntax` and `LeanLean.Kernel` and replays their closures.  Module-closure root selection excludes generated support declarations such as match helpers, no-confusion helpers, constructor eliminators, sparse-case helpers, derived instance support, generated representation declarations, and non-recursive aux recursors.

The self-check closure keeps a narrow trusted base for imported definitions.  It replays recursive aux support for `below` and `brecOn`, including the associated `.go` and `.eq` support names, because local structurally recursive source definitions compile through those declarations.  Other generated support declarations enter as typed dependencies unless selected as explicit roots, so the self-check claim concerns source-facing kernel code and the recursive aux declarations needed to type-check it.

The export-backed self-check script exports the same named source-facing roots from `LeanLean.Syntax` and `LeanLean.Kernel` with `lean4export`, then checks the resulting artifacts with `leanlean-check-export`.  This path does not use Lean's module environment while checking the artifact.  It is a stronger interface check for the named roots, but it does not change the generated-support boundary.

The generated-support inventory mode reports skipped self-check declarations without admitting them.  It classifies each skipped declaration by module, generated-support class, declaration kind, dependency shape, dependency count, and current replay outcome if selected as a root.  This inventory is diagnostic evidence for Phase 13; it is not an admission rule.

`leanlean-check-export` is the first independent artifact checker.  It reads a `lean4export` NDJSON file from an explicit file path or from the Arena `IN` environment variable, parses the accepted export fragment, translates the records into declaration-script entries, and replays those entries through the local kernel.  It does not load the Lean module or inspect Lean's in-process environment while checking the artifact.

The accepted export fragment follows `lean4export` format 3.1.0.  It accepts `Name.str` and `Name.num`; universe levels `succ`, `max`, `imax`, and `param`; expressions `bvar`, `sort`, `const`, `app`, `lam`, `forallE`, `letE`, `proj`, `natVal`, `strVal`, and `mdata`; and declarations `axiom`, `def`, `thm`, `opaque`, `quot`, and grouped `inductive` records.  The parser erases binder annotations, `let` nondependence flags, and expression metadata, because the local term grammar gives those fields no typing or conversion role.

Inductive export records become one kernel-style inductive block followed by generated-constructor and generated-recursor checks.  A generated recursor check includes the exported block-member list, parameter count, index count, motive count, minor-premise count, rule constructor names, field counts, and rule right-hand sides.  Quotient export records coalesce into the local low-level quotient primitive package, while unsafe and partial declaration records remain unsupported at the trusted replay boundary.

Both checker executables return exit code `0` for accepted artifacts, `1` for local-kernel rejection after translation, `2` for unsupported input, and `3` for internal executable failure.  Safe recursive source definitions are accepted when their compiled closures recheck locally.  Unsafe declarations, partial declarations, unknown roots, and untranslated Lean expression forms remain unsupported at these boundaries.

## Lean 4 Divergence Boundary

Lean 4 remains the comparison implementation, while this specification determines which local declarations the checker trusts.  A divergence in this table means that Lean 4 accepts, stores, reduces, or records more than the current subset does, or that the local importer rejects a Lean environment artifact that Lean permits.  Each entry names the local rule and the reason for keeping that boundary.

| Feature | Local rule | Lean 4 behavior or data | Reason |
| --- | --- | --- | --- |
| Source-recursion metadata | The Lean importer accepts safe compiled recursive definitions as ordinary definitions when their closures recheck, and it rejects `mutualDefnDecl` records and partial-definition artifacts. | Lean accepts terminating recursive source definitions and elaborates them into ordinary kernel declarations plus auxiliary metadata. | The subset checks the compiled kernel object, but it does not reconstruct structural recursion, well-founded recursion, equation metadata, or partial-recursion artifacts from source-level metadata. |
| Unsafe declarations | Trusted Lean replay rejects Lean records whose declaration data carries `isUnsafe`. | Lean environments may contain unsafe axioms, definitions, opaque declarations, inductives, generated constructors, or generated recursors. | The safe fragment has no soundness account for unsafe declarations. |
| Full string-literal computation | String literals type-check as `String` and remain neutral under local normalization. | Lean reduces string literals through the prelude's string, character-list, and byte-array machinery. | The subset has specified raw string literals but has not specified the recursive and primitive support needed for Lean's full string computation path. |
| Kernel-overridden primitive computation | The local conversion rule includes only the named primitive reductions listed in the conversion section: `Nat.add`, `Nat.mul`, `Nat.pow`, `Nat.sub`, `Nat.beq`, and `Nat.ble`. | Lean's prelude contains declarations whose exported logical models are accompanied by kernel or compiler overrides.  Exported proofs can rely on those definitional equalities even when the theorem statement does not mention the primitive directly. | Each override must enter through an explicit table entry, source citation, implementation rule, and comparison test.  Unlisted overrides remain unsupported rather than being approximated by ad hoc unfolding behavior. |
| Higher-level quotient API | The kernel trusts only `Quot`, `Quot.mk`, `Quot.lift`, `Quot.ind`, and `Quot.sound` as low-level quotient primitives. | Lean's library exposes `Quotient`, setoid notation, and derived quotient eliminators. | The higher-level API can be defined or imported after the low-level primitives receive a complete account.  It should not enter as an unspecified primitive. |
| Arbitrary Lean modules and environment fragments | Root-name closure extraction, replay, and differential comparison cover named finite fragments. | A Lean module can contain declarations that depend on recursive artifacts, unsafe declarations, broader strings, elaborator metadata, or library definitions outside this subset. | Full module replay requires a written rule for every dependency class.  The named fragments provide comparison evidence, not a general admission rule. |
| Generated support declarations outside recursive aux support | Self-check module-closure mode excludes generated match helpers, no-confusion helpers, constructor eliminators, sparse-case helpers, derived instance support, generated representation declarations, and non-recursive aux recursors as roots.  It treats them as typed dependencies unless selected explicitly. | Lean stores many generated support declarations as ordinary constants, abbreviations, or auxiliary recursors in the environment.  Some of those declarations rely on library wrappers, sparse-case compilation, or primitive operations outside the current self-check claim. | The current self-check claim is about source-facing kernel declarations and recursive `below`/`brecOn` support.  A complete drop-in checker needs a generated-support specification for every generated declaration class it accepts as a checked value. |
| Structure elaboration metadata | The importer reads direct field order, direct field projection functions, subobject parents, and direct parent projection metadata. | Lean's structure extension also records defaults, resolution order, class-instance information, notation-facing data, deprecated auto-parameter fields, and binder annotations. | The omitted fields guide elaboration and notation rather than kernel typing or conversion in the current subset. |
| Sort-polymorphic inductive results | The inductive checker accepts `Prop`, definitely data-valued results, empty sort-polymorphic results, and one-constructor no-field sort-polymorphic results. | Lean's prelude contains `PEmpty` and `PUnit`, and future comparisons may expose additional sort-polymorphic shapes. | The current rule characterizes only the shapes already needed for replay while preserving rejection for data-carrying ambiguous `Sort u` declarations. |
| Structure eta outside the current projection fragment | Eta conversion applies only to non-recursive, data-valued, one-constructor inductives whose constructor target matches the inferred structure type. | Lean has structure eta behavior tied to its projection and structure machinery.  The corpus already preserves rejection for computed-index eta. | The local eta rule stays narrow until recursive structures and more indexed structure shapes have a complete soundness argument. |

These boundaries are part of the trusted specification.  The checker implements a listed feature, rejects it at the stated boundary, or leaves it to a later phase with no admission path.  Later versions may enlarge the table only by adding a rule, implementation coverage, and comparison tests.
