# Faithfulness Harness

The faithfulness harness compares the local kernel fragment against observed Lean 4 behavior on small source examples.  It does not make Lean 4 the project specification, because the local specification remains the authority for this repository.  It records agreement and disagreement with Lean 4 as evidence, and every resulting decision still needs a written rule in `spec.md`.

The first corpus lives under `Faithfulness`.  `Accepted.lean` contains Lean source that the installed Lean compiler must accept, including universe-polymorphic definitions, theorem declarations, inductive recursors, proposition-valued recursors, equality elimination, low-level quotients, mutual inductives, nested mutual inductives, indexed structure projections, inherited structures, recursive source definitions, transparent definitions, abbreviations, opaque constants, literal declarations, lifts, dependent pairs, and subtypes.  `ModuleA.lean` and `ModuleB.lean` add a module-boundary import case with a recursor over an imported inductive.  The files under `Faithfulness/Rejected` contain Lean source that the installed Lean compiler must reject, including a non-positive inductive, a theorem whose type is not a proposition, an ambiguous `Sort u` inductive result, forbidden eliminations from propositions into data, invalid projections, failed eta across computed indices, an opacity-sensitive `rfl`, a quotient relation mismatch, and a nested inductive parameter that contains a local variable.

The executable `lake exe leanleanfaith` runs the corpus through the installed `lean` command.  `Accepted.lean` and `ModuleB.lean` must exit successfully, and each rejected file must fail with an expected diagnostic fragment.  The environment variable `LEANLEAN_LEAN` can name a specific Lean binary when the default command does not select the intended compiler.

The local bridge lives in the ordinary regression suite, `lake exe leanleanimport`, and `lake exe leanleandiff`.  The regression suite checks corresponding local expressions for the same behavioral classes, including raw literal translation, natural-literal computation, dependent family parameters, supported sort-polymorphic subsingletons, and recursor generation over declared field types.  The importer smoke test loads selected roots from the accepted corpus, module-boundary roots, and a named core set through Lean's environment, extracts their `ConstantInfo` closures, reads Lean's structure extension for structure roots, and replays the result through the local declaration importer.  The smoke roots now list structures without listing their field projection functions, so the test checks that structure-extension metadata pulls those projection declarations into the closure.  The same smoke file checks safe compiled recursive definitions from the local corpus and core roots such as `Nat.add` and `List.map`.  The differential executable elaborates selected Lean terms, translates Lean's inferred types and reduced values, and checks the local inferred types and normalized values against them.  It also replays named finite environment fragments and compares translated constant types for every imported constant and reduction behavior for imported transparent definitions.  Together these paths include ordinary definitions, safe compiled recursive definitions, opaque definitions, propositions, mutual inductives, nested mutual inductives, inherited structures, projections, natural literals, equality, quotients, `Sigma`, `PSigma`, `PEmpty`, `PUnit`, `Unit`, `ULift`, `PLift`, `Subtype`, `Decidable`, core inductives, and indexed local inductives.  Kernel-overridden primitive computation is now tracked as a separate comparison class: `Nat.add`, `Nat.mul`, `Nat.pow`, `Nat.sub`, `Nat.beq`, and `Nat.ble` are specified cases, and future accepted export tests should force any newly admitted primitive through an exported proof rather than relying only on local normalization examples.  String literals currently have source-level acceptance coverage, but they are absent from the importer smoke roots because the full `String` environment closure reaches recursive string and byte-array definitions outside the specified subset.

`lake exe leanlean-check-module` exposes the environment-closure replay path as a command-line checker for compiled Lean modules.  It takes `--module <module>` plus one or more `--decl <name>` roots, loads the module, extracts the root closure, and replays that closure through the local kernel.  This executable remains useful for diagnosis because it compares local replay against Lean's loaded environment.

The module checker reports unsupported declarations when the loaded environment records unsupported kernel data, such as an unsafe definition.  It does not classify roots by source syntax after Lean compiles the module: in Lean 4.29.1, a simple `partial def` in the local unsupported test module appears as a safe opaque constant, and the module checker therefore checks it as an opaque declaration.  Explicit partial records in exported artifacts remain unsupported by `leanlean-check-export`.

`lake exe leanlean-self-check` specializes the module-checker path to this repository's compiled kernel modules.  The default run checks the named kernel-only slice, and `--module-closure` checks source-facing declarations from `LeanLean.Syntax` and `LeanLean.Kernel`.  The module-closure mode excludes generated support roots and replays only the recursive `below` and `brecOn` aux support needed by those source-facing declarations, so it is evidence about the kernel source slice rather than a claim that every generated Lean support declaration has a local specification.

`tools/export-self-check.sh` checks the same named source-facing roots through the independent export artifact path.  It uses `lean4export` to produce one artifact for the selected `LeanLean.Syntax` roots and one artifact for the selected `LeanLean.Kernel` roots, then runs `leanlean-check-export` on both artifacts.  This script checks the artifact interface for the named roots; it does not replace the generated-support work needed for full compiled-module self-check coverage.

`lake exe leanlean-self-check --generated-inventory` reports skipped generated-support declarations by class, declaration kind, dependency shape, and replay outcome.  It currently inventories 466 skipped declarations from `LeanLean.Syntax` and 1297 from `LeanLean.Kernel`.  The output is diagnostic: it identifies which generated classes already replay and which fail or remain unsupported when selected as roots.

`lake exe leanlean-check-export` checks the independent `lean4export` NDJSON artifact.  It reads either an explicit artifact path or the Arena `IN` environment variable, parses the specified export fragment, and replays the resulting declaration script without loading the Lean module.  The generated accepted tests live in `tools/export-pipeline-tests.sh`.  With `LEANLEAN_LEAN4EXPORT=/tmp/lean4export/.lake/build/bin/lean4export`, the script builds the source modules, runs `lean4export` on selected roots, and accepts the resulting artifacts.  The cases cover a custom inductive recursor, a theorem declaration, an opaque declaration, a raw natural literal, closed natural subtraction, quotient primitives, a subtype value, a compiled recursive list definition, and the gcd/parity arithmetic theorem.  The same script now requires rejection for two generated Arena-derived source fixtures, plus static Arena copies for an `imax` normalization attack and a bad constant-level unfolding attack.  Each rejection case checks a diagnostic fragment as well as the exit code.  The gcd/parity artifact currently checks 572 declaration entries.  The earlier `omega` proof variant remains a separate broad-performance stress case because it reaches `Lean.Omega` certificate dependencies outside this fixture's purpose.

Both command-line checkers use the same outcome vocabulary.  Accepted roots or artifacts return exit code `0`, unsupported roots or artifacts return exit code `2`, and local replay rejection after translation returns exit code `1`.  Current realistic accepted module-checker examples include `LeanLeanFaithfulness.Accepted.transparentId`, `LeanLeanFaithfulness.Accepted.Pair`, `LeanLeanFaithfulness.Accepted.natDouble`, `LeanLeanFaithfulness.Accepted.listLength`, and the mutual pair `LeanLeanFaithfulness.Accepted.evenFlag` and `LeanLeanFaithfulness.Accepted.oddFlag`.
